# Design

## Overview

A simple webpage (HTML, some CSS themes, plain JS with extra libraries as needed), deployed to GitHub Pages. It should look nice and modern, and have graphs. Reaching for a JS/HTML framework is fine if it gets complicated — but start plain, since it probably won't.

There is no backend, so **all calculations happen client-side**. This is a feature, not just a constraint: your interval usage data never leaves your browser. The page should say so plainly.

Utility and generation rates are read from JSON files served out of `rates/`.

First utility: **SDG&E**. First two CCA generation providers: **San Diego Community Power (SDCP)** and **Clean Energy Alliance (CEA)**.

## Core billing model

This is the rule everything else depends on:

```text
total = SDG&E delivery
      + fixed charges
      + CCA adders (CCA customers only)
      + generation  ← the only part that differs by provider
```

Switching to a CCA does not replace your bill. SDG&E still delivers the power and still bills you for delivery. The CCA replaces only the **generation** line item, and CCA customers additionally pay SDG&E-set adders (PCIA, franchise fee, DWR bond) that bundled SDG&E customers don't.

Modeling each of the three providers as a standalone flat rate would produce comparisons that look plausible and are wrong. The schema below enforces the split structurally.

## Rate file schema

Every rate file uses the same format so the frontend can read them all through one code path.

### `rates/index.json` — manifest

GitHub Pages serves no directory listings, so the app cannot discover what's in `rates/` at runtime. A committed manifest is the index. Generated by the rate extractor (below), not hand-maintained.

```json
{
  "generated": "2026-07-18",
  "files": [
    { "provider": "sdge", "type": "utility",    "effective_date": "2026-06-01", "path": "sdge.json" },
    { "provider": "sdcp", "type": "generation", "effective_date": "2026-06-01", "path": "cca-sdcp.json" },
    { "provider": "cea",  "type": "generation", "effective_date": "2026-06-01", "path": "cca-cea.json" }
  ]
}
```

Rates change several times a year. The app picks the file whose `effective_date` covers the usage period, and warns visibly when the newest available rates predate the usage data.

### `rates/sdge.json` — base utility file

One base file holds everything SDG&E sets: delivery, fixed charges, baseline, calendar definitions, CCA adders, and SDG&E's own generation prices.

```json
{
  "provider": "sdge",
  "effective_date": "2026-06-01",
  "source_url": "https://…",
  "verified_against": "SDG&E Schedule TOU-DR1, sheet 3",

  "fixed_charges": {
    "daily_service_charge": 0.0,
    "minimum_bill_daily": 0.0
  },

  "baseline": {
    "climate_zones": {
      "coastal": { "summer_kwh_per_day": 0.0, "winter_kwh_per_day": 0.0 }
    },
    "credit_per_kwh": 0.0
  },

  "seasons": {
    "summer": { "start": "06-01", "end": "10-31" },
    "winter": { "start": "11-01", "end": "05-31" }
  },

  "day_types": {
    "weekday": "mon-fri, excluding holidays",
    "weekend": "sat-sun",
    "holiday": ["2026-01-01", "2026-07-04", "…"]
  },

  "cca_adders": {
    "pcia_per_kwh": 0.0,
    "franchise_fee_per_kwh": 0.0,
    "dwr_bond_per_kwh": 0.0
  },

  "plans": [
    {
      "id": "tou-dr1",
      "name": "TOU-DR1",
      "eligibility": { "notes": "recorded, not enforced yet" },
      "delivery": {
        "summer": {
          "weekday": [ { "start_hour": 16, "end_hour": 21, "price_per_kwh": 0.0 } ],
          "weekend": [ { "start_hour": 16, "end_hour": 21, "price_per_kwh": 0.0 } ]
        },
        "winter": { "…": [] }
      },
      "generation": {
        "summer": {
          "weekday": [ { "start_hour": 16, "end_hour": 21, "price_per_kwh": 0.0 } ]
        }
      }
    }
  ]
}
```

Hour blocks must tile the full 24 hours with no gaps or overlaps — worth validating on load, since a gap silently under-bills.

### Two pricing models

Not every residential plan is time-of-use, so `pricing_model` on each plan selects the shape:

- `"tou"` (default) — `season → day_type → hour blocks`, as above.
- `"tiered"` — `season → tiers`, priced by cumulative consumption against the baseline allowance with no time-of-day component. No day_type level, since the day of week doesn't affect the bill.

```json
"delivery": {
  "summer": [
    { "up_to_pct_of_baseline": 130, "price_per_kwh": 0.22876 },
    { "up_to_pct_of_baseline": null, "price_per_kwh": 0.33539 }
  ]
}
```

The last tier is always `null` (open-ended); a capped final tier leaves the heaviest usage unpriced.

This matters more than it looks: SDG&E's **DR** is the default residential schedule and is tiered, so it's the plan most users are comparing *away from*. Omitting it would leave out the baseline of the whole comparison.

Costing a tiered plan needs the billing period, not just the interval series — the tier boundary is `allowance_kwh_per_day × days_in_period × 1.30`, and it depends on climate zone. A tiered plan therefore cannot be scored from a partial period as cleanly as a TOU plan can.

### `rates/cca-sdcp.json`, `rates/cca-cea.json` — generation overlays

CCA files are **generation-only**. They never restate delivery, fixed charges, or adders, because CCAs don't set those.

```json
{
  "provider": "sdcp",
  "type": "generation",
  "effective_date": "2026-06-01",
  "source_url": "https://…",
  "plans": {
    "tou-dr1": {
      "summer": {
        "weekday": [ { "start_hour": 16, "end_hour": 21, "price_per_kwh": 0.0 } ]
      }
    }
  }
}
```

Keys under `plans` match `plans[].id` in `sdge.json`. A CCA that doesn't serve a given SDG&E plan simply omits it. Where a CCA offers multiple products (base/green tiers), each is a separate file, distinguished by `product`.

**A CCA can publish more than one rate schedule.** SDCP publishes two, split by enrollment year: one for San Diego, Chula Vista, Encinitas, Imperial Beach and La Mesa, another for National City and the unincorporated county. Same product name, different prices. The enrollment year is also the customer's PCIA vintage, so the two travel together — picking the wrong group gets both the generation rate and the exit fee wrong.

So the UI's provider question is really two questions: **which CCA**, and **which product**. For SDCP the rate group follows from the city, which the user already effectively states. Files carry `rate_group`, `service_area`, and `pcia_vintage` so this can be driven from data rather than hardcoded.

A `generation_credit_per_kwh` field holds any per-kWh credit applied on top of the published prices (CEA currently runs one). It stays a separate field rather than being folded into the prices, so the numbers remain checkable against the published schedule and a lapsed credit doesn't quietly become permanent.

## Location selection

**v1: the user picks their climate zone directly** from a four-option dropdown — Coastal, Inland, Mountain, Desert. The zone drives the baseline allowance, and that is the only thing the calculator needs location for.

No zip lookup, and no `zips.json`. The reason is that SDG&E doesn't publish one: climate zones are defined by "Territory Served, Map No. 1", which is a raster map of geographic bands labeled by city, with no zip codes anywhere on it. Deriving a mapping by eye would produce confidently wrong baseline allowances.

This costs the user very little, because **their bill prints the zone** (`Climate Zone: Coastal`). Link the map next to the dropdown for anyone without a bill to hand.

The CCA provider is likewise a direct choice rather than something inferred from location.

**Later**, if a zip lookup is wanted: SDG&E's baseline allowance calculator does resolve zip to zone server-side, so a mapping could be built by querying it. That's a data-collection task with a verification burden, not a v1 feature.

## Green Button import

SDG&E exports interval data as either CSV or ESPI (Green Button) XML. Accept both, choosing the parser from the uploaded file — sniff the content rather than trusting the extension alone.

Both parsers normalize into one internal series:

```js
[ { start: Date, duration: seconds, kWh: number } ]
```

Everything downstream sees only the normalized series and never knows which format it came from. Assume 15-minute intervals, handle hourly. A year of 15-minute data is ~35,000 intervals — fine in-browser.

## Calculator

1. **Which TOU rate is best for my situation** — score every plan against the actual interval series, rank by cost over the selected period.
2. **How much each plan would cost me** — itemized breakdown showing delivery / generation / fixed / adders as separate lines, so the CCA comparison is legible rather than a single opaque total.
3. **If I added a specific load, how would that change things** — the added load is an hourly *shape*, not a monthly kWh total. Charging an EV at 1am and at 6pm produce opposite conclusions from the same kWh.

**Eligibility is deferred.** Every plan is scored for now. The `eligibility` field is recorded in the schema and surfaced as an unenforced note, so filtering can be turned on later without touching the rate files.

### Analysis period

A selector controlling which slice of the imported data gets costed:

- **Year to date** — Jan 1 through the end of the data.
- **Full export** — everything the export covers. Label it from the actual data rather than hardcoding a period, since exports vary: "Last 13 months (Jun 2025 – Jul 2026)".

Custom start/end dates are a later addition if wanted; these two cover the common cases.

**Incomplete days are trimmed by default, with a toggle to keep them.** Green Button exports typically start and end mid-day, and a partial day drags down averages and misstates the daily-shape graph. The user can turn trimming off to see the raw import.

Detecting an incomplete day means counting intervals against what the day should hold — but **do not assume 96 intervals (or 24 hourly)**. DST transition days genuinely have 23 and 25 hours, so a naive count flags two legitimate days per year as incomplete. Compute the expected count from the day's actual length in the local timezone.

### Period coverage affects correctness

Seasonal pricing means a partial year is not a small version of a full year — summer peak rates dominate the annual answer, so a winter-only export ranks plans differently than a full year would.

Therefore:

- Cost is reported **for the selected period**, not silently annualized.
- Annualizing is offered only as an explicit, labeled estimate.
- When the selected period doesn't span a full year, **warn visibly**, and say which seasons are missing. "Based on 4 months (winter only) — summer peak rates aren't represented" is the difference between a useful tool and a confidently wrong one.

### Graphs

The charts are how the result gets understood, so they matter as much as the math:

- **Daily load shape** — average usage by hour, with TOU peak windows shaded, so it's visually obvious whether usage lands in peak.
- **Plan comparison** — annual cost per plan, stacked by delivery / generation / fixed / adders.
- **Seasonal usage** — monthly totals, since summer peak pricing dominates the annual answer.
- **Added-load before/after** — the same daily shape with the new load overlaid.

## Load profiles

`profiles/` holds JSON load profiles, listed in `profiles/index.json` (same reason as the rates manifest).

```json
{
  "id": "ev-overnight",
  "name": "EV charging, overnight",
  "annual_kwh": 3000,
  "hourly_shape": [0.08, 0.08, "…24 values summing to 1.0"],
  "notes": "Level 2, midnight–6am"
}
```

Ships with a starter library: EV charging on a few schedules, heat pump, pool pump.

Users can also define their own in the page. Those live in localStorage and are **exportable as this exact format**, so a good user-built profile can be contributed back to the repo as a file with no conversion.

## Rate extractor

An **occasional offline LLM skill** (`.claude/skills/`), run by hand when rates change. Not part of the deployed site, not a runtime feature.

Given a utility rate PDF or rate webpage, it:

1. emits a rate JSON conforming to the schema above,
2. regenerates `rates/index.json`,
3. records `source_url` and `effective_date` on the output.

Output is a **draft requiring human review before commit**. Unreviewed extraction output must never reach the calculator — a silently wrong tariff number is worse than no tool at all.

PDF extraction and webpage extraction are different enough that they should be separate paths rather than one abstraction pretending to cover both.

### Triggering a rescrape

Manual to start. Automating it on GitHub (scheduled action) is TBD, and the human-review requirement is the thing that makes it awkward — a fully automatic scrape-and-commit would put unreviewed tariff numbers straight into the calculator. If it does get automated, it should **open a PR rather than commit to main**, so review stays in the loop. A scheduled job that only checks whether the source document changed, and notifies, is a safer middle step.

## Validation

Correctness *is* the product here, so there needs to be a known-answer check.

There is deliberately **no bill import feature**. Validation is a manual step: hand a real SDG&E bill plus the matching interval data to an LLM and confirm the calculated total matches. Confirmed results get recorded in the repo as regression references, so a later rate-schema change that breaks the math is visible.

## Deferred

- **Solar / NEM (phase 2).** NEM 2.0 and NEM 3.0 net billing are structurally different — net billing prices exports against an hourly export-value table rather than netting against retail. Deferred, but the rate schema should leave room for export pricing now, since retrofitting it later would mean rewriting every rate file.
- **Eligibility enforcement.** Schema field exists and is populated; nothing reads it yet.
- **Additional utilities beyond SDG&E.**
